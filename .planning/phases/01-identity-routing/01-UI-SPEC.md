## Typography

> In this domain, "typography" is **content structure** — the markdown elements and section patterns used in each file type. Must be consistent across all CLAUDE.md files in this phase.

| Role | Markdown Element | Usage | Line Height (structural) |
|------|-----------------|-------|--------------------------|
| Title | `# heading` (single H1 per file) | First line of every CLAUDE.md, format: `# path/ — description` | 1 blank line after |
| Section header | XML-like tag `<tag_name>` ... `</tag_name>` | Major content sections: `<what_is_this>`, `<session_protocol>`, `<identity>` | 1 blank line before and after |
| Directory entry | `@dir/` or `@file.md` with inline description | Navigation lists, separated from label by spaces + em-dash | 1 per line, no blank lines between entries |
| Annotation | `- plain text` bullet | Supporting prose within sections | 1 per line |

**Section tags in use (established pattern from personal/CLAUDE.md):**
- `<what_is_this>` — top-level description of the directory's purpose
- `<identity>` — persona inline content (root CLAUDE.md only)
- `<session_protocol>` — behavioral directives (personal/CLAUDE.md only)
- `<core_identity>` — identity shorthand (personal/CLAUDE.md only)
- New files: use `<what_is_this>` for purpose; omit session_protocol unless the directory warrants it

**Weight equivalents (emphasis):**
- Regular (400 equivalent): `@dir/` entries with descriptions — plain markdown
- Semibold (600 equivalent): Section tag names and H1 titles — structural anchors only

## Color

> In this domain, "color" is **loading behavior** — whether content is eager (immediately in context) or lazy (deferred until needed). The 60/30/10 split maps to context budget allocation.

| Role | Loading Mode | Elements | Context Budget Impact |
|------|-------------|----------|----------------------|
| Dominant (60%) | Lazy — `@dir/` tree preview | All directories in root CLAUDE.md; soul/ subdir; memory-blocks/ | Minimal: shows structure without loading content |
| Secondary (30%) | Conditionally eager — `@file.md` for short, high-value files | personal/profile.md (4.5K, always needed for identity card) | Moderate: short file, high value, loaded once |
| Accent (10%) | Eager cascade — `@dir/CLAUDE.md` hard-link | Only files that MUST be in context from absolute session start | High: cascades into full file loading |

**Accent (eager load) reserved for:** profile.md in personal/CLAUDE.md only. No other files in Phase 1 qualify. The soul document (29K) must never appear as an eager link in any CLAUDE.md.

**Destructive equivalent — eager injection of large files:**
- Flagged pattern: `@soul-document.md` in personal/CLAUDE.md (currently present — must be removed)
- Replacement: `@soul/` tree preview (lazy)
- Any new `@file.md` link over 10K in a router = must be justified or converted to `@dir/` link

## Copywriting Contract

> In this domain, "copywriting" is the prose inside CLAUDE.md files — the descriptions, purpose statements, and navigation labels Claude reads to orient itself.

| Element | Contract |
|---------|---------|
| Directory entry label | Format: `@path/` + 4-10 spaces + em-dash + 1 brief noun phrase (max 8 words). Example: `@soul/          — foundational identity, split into sections` |
| H1 title | Format: `# path/ — one-line purpose`. Never just `# path/` with no description. |
| What-is-this body | 1-3 sentences maximum. Answers: what lives here, who it's for, what time span it covers. |
| Empty directory state | If a directory is empty or near-empty: root CLAUDE.md notes it as `@state/    — empty, pending Phase 2`. Never omit it; never create a CLAUDE.md for a truly empty directory. |
| Tier label (memory-blocks) | Must contain one of: "LORE", "museum", "reference only", "not directives". Must NOT imply active behavioral constraints. |
| Session protocol | Only in personal/CLAUDE.md. 4-6 bullet points. Each point is a directive verb phrase ("read ...", "trust ...", "ask before ..."). No prose. |
| Last validated date | Required in personal/soul/CLAUDE.md and personal/core/CLAUDE.md. Format: `**Last validated:** YYYY-MM-DD (what was checked)` |
| Error/404 state | A CLAUDE.md that references a nonexistent file is a broken link. Preventive: verify every `@` link resolves before committing the file. |
| Destructive action | Removing an eager `@file.md` link from personal/CLAUDE.md. Confirmation approach: grep for the removed link in any other CLAUDE.md to ensure the file remains reachable via another path. |

## Information Architecture

> The complete navigation structure this phase must produce. Every node is a deliberate routing decision.

### Two-Hop Guarantee

Every artifact in the home must be reachable in two deliberate choices from root.
- Hop 0 (free): Root CLAUDE.md always in context at session start
- Hop 1: Navigate to a directory — its CLAUDE.md auto-loads on file read
- Hop 2: From that CLAUDE.md, follow one `@link` to reach the target artifact

The soul directory at personal/soul/ is 3 path segments deep but only 1 deliberate choice (enter personal/soul/).

## Navigation Patterns

> The `@`-link strategy is the core interaction model. Each pattern has a specific use case.

| Pattern | Syntax | Load Behavior | When to Use |
|---------|--------|---------------|-------------|
| Lazy directory | `@dir/` | Tree preview only — no CLAUDE.md loads | Root routing to all subdirectories; default for all new entries |
| Conditionally eager | `@file.md` | File content injected immediately | Only profile.md in personal/CLAUDE.md — short, identity-card, always needed |
| Eager cascade | `@dir/CLAUDE.md` | Loads CLAUDE.md immediately + any of its own `@` links | Not used in Phase 1 — no file warrants this |
| Inline content | Plain prose in CLAUDE.md | Always in context | Root identity block; session protocol; what-is-this descriptions |

**Maximum import depth:** 5 hops (Claude Code limit). Phase 1 target: maximum 2 active hops from root.

## Scannability Contract

> How Claude parses and extracts information from a CLAUDE.md. Measured in "time to locate" — how many lines must be read before Claude knows where to navigate next.

| Rule | Contract |
|------|---------|
| Navigation list position | Always the first or second section after the H1, never buried after prose |
| Description completeness | Every `@link` entry must have a description. `@soul/` alone is not sufficient — must say what's there |
| Prose limits | No more than 3 sentences of prose outside of a tagged section. Navigation lists are not prose |
| Tag usage | Use XML-like tags for sections; do not use H2/H3 headings for sections inside a CLAUDE.md |
| Ordering | Most frequently needed artifacts first in navigation lists |

---

## Consistency Contract

> Rules that apply identically to every CLAUDE.md file in this phase.

| Rule | Requirement |
|------|------------|
| H1 format | `# path/ — purpose` where path matches the actual filesystem path |
| Entry format | `@link` + at least 4 spaces + `—` + description (no colons, no periods at end) |
| Tag names | Lowercase with underscores: `<what_is_this>`, `<session_protocol>`, `<identity>` |
| Line count | Hard maximum 50 lines per file (CTX-03). Target is well under — see spacing scale |
| Relative paths | All `@` links use relative paths. No absolute paths. No `/home/skogix/` |
| Last validated | personal/soul/CLAUDE.md and personal/core/CLAUDE.md must include `**Last validated:** YYYY-MM-DD` |
